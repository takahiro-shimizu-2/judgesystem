# judgesystem マスター・保存先 棚卸し

**対象**: 非エンジニア向け  
**目的**: 「どんなマスターが必要か」「主な項目は何か」「どの資源がどこに保存されるか」を1つにまとめて把握できるようにする

---

## 1. まず結論

このプロジェクトでは、必要なマスター一覧、各マスターの項目、保存先の情報が**1か所にはまとまっていません**。  
そのため、この文書では、現在のコードと文書を突き合わせて、業務上わかる形に整理しています。

非エンジニア向けに言うと、次のように理解すれば十分です。

- `マスター` = システムが判断や表示の前提として持っておくべき基本台帳
- `業務データ` = 公告、要件、判定結果、進捗、候補会社など日々増える情報
- `資源` = PDF、Markdown、OCR JSON、添付ファイルなどのファイル類
- `保存先` = PostgreSQL、Google Cloud Storage、ローカルファイル、Google Sheets など

---

## 2. この文書の情報源

この文書は、次の優先順位で整理しています。

### 信頼度が高いもの

1. `db/migrations/`  
   現在のDBに近い定義がある

2. `db/seeds/seed.sh`  
   どのマスターをどう投入するかが分かる

3. `packages/backend/src/config/database.ts`  
   Backend が実際にどのテーブルを使うかが分かる

4. `packages/engine/` の実装  
   PDFやOCR JSONの保存先、GCS利用の実態が分かる

### 参考にはなるが古い可能性があるもの

5. `doc/master_description/`  
   項目の意味は詳しいが、現在のスキーマとズレている可能性がある

---

## A. 企業・拠点・資格系

### 1. `company_master`

**何のマスターか**  
企業そのものの基本情報と、欠格要件の判定に使うフラグを持つ台帳

**主な項目**

- 企業番号
- 会社名
- 法人番号
- 設立日
- 住所
- 電話
- 代表者名
- 主業種
- 破産・更生・反社会勢力・社会保険などの各種判定フラグ
- 備考

**何に使うか**

- 欠格要件チェック
- 判定結果とのひも付け

**何のマスターか**

- 拠点種別
- 拠点住所
- 電話
- メール
- 所在都道府県

**何に使うか**

- 所在地要件チェック
- 拠点単位の判定
- 画面表示

### 3. `office_registration_authorization_master`

**何のマスターか**  
拠点がどの発注機関・工種・等級で登録されているかを管理する台帳

**主な項目**

- 拠点番号
- 発注機関番号
- 工種番号
- 等級
- スコア
- ステータス
- 登録日
- 有効期限
- 指名停止状態

**何に使うか**

- 等級・登録要件の判定
- 指名停止などの確認

### 4. `office_work_achivements_master`

**何のマスターか**  
拠点ごとの工事実績を持つ台帳

**主な項目**

- 実績番号

- 発注機関番号

### 5. `technician_qualification_master`

**主な項目**

- 資格番号

- 種別

### 6. `employee_master` / `employee_qualification_master` / `employee_experience_master`

**何のマスターか**  
従業員、資格、経験の台帳

**主な項目**

- 従業員番号
- 所属企業 / 拠点
- 氏名
- 生年月日

**重要な注意**

これらは Engine の処理説明や `master_description` には出てきますが、  
現時点の `database-schema.md` では主一覧として前面には出ておらず、最新DB定義とのズレ確認が必要です。

つまり、**業務上は必要そうだが、現行Webアプリの最新DB定義としては再確認が必要な領域**です。

## 4. 保存先はどこか

ここでは、「何がどこに保存されるか」を業務目線で整理します。

## A. PostgreSQL に保存されるもの

### 1. 業務データ本体

保存先: **PostgreSQL**  
本番想定: **Cloud SQL 上の PostgreSQL**  
ローカル開発: **Docker 上の PostgreSQL**

ここに保存されるもの:

- 公告
- 要件
- 日付情報
- 判定結果
- 充足 / 不足要件
- 協力会社候補
- 担当者
- ワークフロー状態
- 各種マスター

つまり、**表形式の業務データの中心は基本的に PostgreSQL** です。

### 2. 協力会社ファイル

保存先: **PostgreSQL**

`evaluation_partner_files` では、ファイル本体が `BYTEA` でDBに入ります。  
つまり、ここはGCSではなく**DBの中にそのまま保存**する設計です。

保存される主な情報:

- evaluation_no
- partner_id
- 送信 / 受信区分
- ファイル名
- content_type
- サイズ
- バイナリ本体

---

## B. Google Cloud Storage に保存されるもの

保存先: **Google Cloud Storage (GCS)**  
主な用途: Engine が取得・生成したファイル資源

ここに保存される可能性があるもの:

- 公告PDF
- Markdown
- OCR JSON

実装上は、次のようなパスが使われます。

- `gs://ann-files/pdf/...`
- `gs://ann-files/markdown/...`
- `gs://ann-files/ocr_json/...`

### GCSを使う意味

- 大きいファイルをDBに直接持たないため
- Backend からも取り出せるようにするため
- ファイル保存をローカルディスクに閉じないため

### DBとの関係

ファイル本体は GCS にあり、  
**その保存先パスだけを DB の `announcements_documents_master` に持つ** 形です。

主なパス項目:

- `save_path`
- `markdown_path`
- `ocr_json_path`

---

## C. ローカルファイルに保存されるもの

### 1. 開発時のPDF / Markdown / OCR JSON

保存先: **ローカルファイル**

例:

- `output/pdf/...`
- `output/markdown/...`
- `output/ocr_json/...`

これは主に、GCSを使わないローカル実行時の保存先です。

### 2. マスターデータ元ファイル

保存先: **リポジトリ内の `data/master/`**

ここに置かれるもの:

- `company_master.txt`
- `office_master.txt`
- `agency_master.txt`
- `construction_master.txt`
- `technician_qualification_master.txt`
- `partners_master.csv`
- `partners_categories.csv`
- そのほか seed 対象の txt/csv

この `data/master/` は、  
**DBに投入する前の元データ置き場** です。

---

## D. Google Sheets に一時的にあるもの

### 協力会社データの取得元

保存元: **Google Sheets**

`db/scripts/fetch_partners_from_sheets.py` により、Google Sheets から協力会社データを取得し、  
その結果を `data/master/partners_master.csv` などに書き出す流れがあります。

つまり、協力会社データは次の流れです。

```text
Google Sheets
  ↓
data/master/*.csv
  ↓
seed.sh
  ↓
PostgreSQL
```

## 5. どの資源がどこにあるかを一言でまとめると

### 表データ

- 保存先は **PostgreSQL**

### 大きめの文書ファイル

- 保存先は **GCS** または **ローカルファイル**
- 位置情報だけ DB に持つ

### 協力会社の添付ファイル

- 保存先は **PostgreSQL の BYTEA**

### マスターの元データ

- 保存元は **`data/master/`**

### 協力会社元データの一部

- もともとは **Google Sheets**

## 6. いま曖昧・不統一な点

### 1. マスター説明書が古い

`doc/master_description/` は、項目の意味は詳しいですが、  
現在のスキーマとズレている可能性があります。

### 2. employee 系マスターの扱いが分裂している

Engine の説明では必要に見える一方、最新DBの見え方では整理が不十分です。

### 3. 命名規則が混在している

DBには `camelCase`、`snake_case`、`PascalCase` が混在しており、  
非エンジニアから見ても「同じ種類の項目なのに名前が揃っていない」状態です。

### 4. 保存先の思想が統一されていない

- 文書ファイルは GCS / ローカル
- 協力会社ファイルは DB

となっていて、資源の扱い方が一貫していません。

---

## 7. 非エンジニアとしてここだけ見ればよいポイント

まず見るべきなのは次の5点です。

1. 本当に必要なマスターは何か
2. そのマスターで足りない項目はないか
3. その情報はDBに置くべきか、ファイルに置くべきか
4. 参考情報と正式情報が混ざっていないか
5. どの情報が元データで、どれが加工後データか分かるか

この5つが見えるだけでも、かなり全体像をつかみやすくなります。

---

## 8. いまのおすすめ

現時点でのおすすめは次の通りです。

- マスター一覧と項目一覧は、今後この文書を起点に1か所へ寄せる
- 最新の真実は `db/migrations/` を基準にする
- `doc/master_description/` は参考資料として扱う
- ファイル資源は将来的に保存方針を統一する
- employee 系マスターは「現行で正式採用か」「今後必要な設計か」を明確にする

この文書は、その整理の出発点として使うのがよいです。
