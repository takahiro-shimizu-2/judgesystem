# judgesystem 再設計 基本方針書

**ステータス**: 提案  
**対象**: プロダクト責任者・運用責任者・開発チーム  
**前提**: 将来を見据えた best practice を優先し、場当たり的な延命ではなく、段階的な再設計を行う

---

## 1. この文書の目的

この文書は、judgesystem を今後も安全に運用し、拡張しやすくするための**正式な設計方針**を定めるものです。

目的は次の3つです。

1. システムの土台に関する判断を先に固定する
2. 個別Issueの修正方針をぶれさせない
3. 「全部作り直す」のではなく、既存資産を活かしながら段階移行する

---

## 2. 基本方針

### 方針1. Backend API を唯一の正規入口にする

今後の標準構成では、**Database に対する業務上の正規な更新責任は Backend が持つ**。

- Frontend は Backend API 経由でのみデータを扱う
- Engine は内部ワーカーとして扱い、最終的には API またはジョブキュー経由で処理する
- Engine と Backend が同じDBを独立に直接更新する状態は、解消対象とする

**意味**  
データの出入り口を1つにし、整合性事故と障害調査コストを下げる。

**関連Issue**  
`#120`, `#112`, `#114`

### 方針2. API仕様をシステム全体の正本にする

型定義の正本は、TypeScriptのコード断片ではなく、**API契約そのもの**とする。

- OpenAPI または JSON Schema を正本とする
- Frontend / Backend / Engine の型は、正本から生成または同期する
- 手書きの重複型定義は縮小・廃止する

**意味**  
画面、API、Engine の間で「同じデータを違う意味で解釈する」事故を防ぐ。

**関連Issue**  
`#117`, `#123`, `#108`, `#111`, `#116`

### 方針3. Database は制約を持つ設計に戻す

Database は単なる保存先ではなく、**整合性を守る仕組みを持つ基盤**として扱う。

- 命名規則は `snake_case` に統一する
- 外部キー、一意制約、NOT NULL、必要なチェック制約を付与する
- 業務番号と内部IDを分ける
- 破壊的な変更は、安全な移行手順とロールフォワード前提で管理する

**意味**  
アプリケーション側のミスをDBで食い止め、将来のスキーマ変更に耐えやすくする。

**関連Issue**  
`#105`, `#106`, `#121`

### 方針4. 本番環境は認証必須・最小権限を標準にする

本番環境の標準は、**認証あり・権限分離あり・秘密情報は外部管理**とする。

- 本番APIは認証必須
- 権限は RBAC で管理する
- 秘密情報は Secret Manager 等で管理する
- 未認証公開やパスワード直書きを標準にしない

**意味**  
事故が起きたときの影響範囲を抑え、運用監査にも耐えやすくする。

**関連Issue**  
`#103`, `#104`

### 方針5. Frontend は「画面表示」と「サーバー状態管理」を分離する

Frontend は巨大ページと場当たりのデータ取得から脱却し、**標準化された取得・更新パターン**へ寄せる。

- ページから直接 `fetch()` を書かない
- APIアクセスは共通クライアントに集約する
- サーバー状態は TanStack Query など標準的な方式で扱う
- 本番コードにモックデータを残さない
- 画面は責務ごとに分割する

**意味**  
同じようなローディング・エラー・更新処理を画面ごとに作らずに済む。

**関連Issue**  
`#109`, `#110`, `#111`, `#115`, `#118`, `#122`

### 方針6. Engine は内部ワーカーとして責務分離する

Engine は巨大関数・巨大モジュールのまま延命せず、**処理単位で分割し、テストできる構成**にする。

- 取得、解析、OCR、保存、判定を分ける
- 例外処理、ログ、待機設定、GCS操作を分離する
- 文字列連結による危険なSQLは廃止する
- 未実装関数やアンチパターンを残さない

**意味**  
Engine の一部修正が全体崩壊につながる状態を減らす。

**関連Issue**  
`#103`, `#112`, `#113`, `#114`

### 方針7. 全面リライトは行わず、段階移行を採用する

今回の再設計では、**既存機能を止めずに、正しい構造へ順番に置き換える**。

- 既存機能をなるべく残す
- 互換レイヤーを使いながら移行する
- いきなり全画面・全機能を置き換えない
- 重要な境界から先に直す

**意味**  
業務停止や大規模な作り直し失敗のリスクを下げる。

## 3. 採用する設計標準

### システム境界

- 利用者は Frontend からのみ操作する
- Frontend は Backend API を唯一の窓口とする
- Engine は内部ワーカーであり、将来的に直接DB更新しない

### API

- OpenAPI ベースで仕様を管理する
- エラーレスポンス形式を統一する
- 入力検証を標準化する

### Database

- 命名は `snake_case`
- 内部IDを主キーとし、業務番号は別カラムで管理する
- 外部キーと整合性制約を積極的に使う

### セキュリティ

- 本番は認証必須
- RBAC 採用
- Secrets はリポジトリ・スクリプトに直書きしない
- 権限は最小権限原則で付与する

### Frontend

- サーバー状態は共通方式に統一
- ページ直書きの API 呼び出しを禁止する
- モックデータは開発専用に限定する

### Engine

- 処理責務を分離する
- SQL は安全な方式に統一する
- 構造化ログを採用する

---

## 4. やらないこと

この方針では、次の進め方は採用しません。

- いきなり全機能を作り直す全面リライト
- Engine と Backend の二重DB更新を恒久的に残す
- TypeScriptだけで型共有したことにしてPython側を放置する
- 本番で未認証アクセスを標準にする
- モックデータやハードコードを本番経路に残す
- DB制約なしでアプリケーション側だけに整合性を押しつける

---

## 5. 成功条件

この再設計が成功したと判断する条件は次のとおりです。

1. Frontend / Backend / Engine のデータ契約が1つの正本から管理されている
2. 本番APIが認証必須で運用されている
3. Engine と Backend の責任境界が明確である
4. DBスキーマに命名規則と制約の標準がある
5. 本番経路にモックデータが残っていない
6. 巨大関数・巨大ページが分割され、主要処理にテスト可能な境界がある
7. 危険度の高いセキュリティ課題が解消されている

## 7. 意思決定ルール

今後、個別修正で迷った場合は、次の順番で判断する。

1. Backend を正規入口にする方針に反しないか
2. API仕様を正本にする方針に反しないか
3. DB整合性を弱める修正になっていないか
4. 本番セキュリティを下げる修正になっていないか
5. 将来の移行を難しくする場当たり対応ではないか

この5つのいずれかに反する場合、その修正は採用しない。

---

## 8. 補足

この文書は、「最小コストで今すぐ動く方法」ではなく、  
**今後も運用・拡張・監査に耐える方法を優先する** ための方針書です。

したがって短期的には遠回りに見える対応もありますが、長期的には修正のやり直しと運用事故を減らすための投資と位置付けます。
