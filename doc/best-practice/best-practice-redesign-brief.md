# judgesystem 再設計方針メモ

**対象**: 非エンジニア向け  
**目的**: 「今のシステムを、将来も安心して使える形にするには何を決めるべきか」を1ページで共有する  
**前提**: 目先の場当たり修正ではなく、将来を見据えた best practice を優先する

---

## 1. 結論

このシステムは、全部を捨てて作り直す必要はありません。  
ただし、**土台の設計は引き直した方がよい** 状態です。

いまは、

- Python Engine と Backend が同じDBを別々に直接操作している
- Frontend / Backend / Engine で型やルールが分かれている
- 認証・認可の標準が弱い
- DBの命名規則や制約が不統一

という状態で、機能追加や修正のたびに「別の場所で事故が起きやすい」構造になっています。

そのため、best practice を選ぶなら、**全面リライトではなく、再設計して段階的に載せ替える** のが最善です。

---

## 2. 目指す姿

将来を見据えた標準形は、次のような構成です。

```text
利用者
  ↓
Frontend
  ↓
Backend API  ← システムの正規入口
  ↓
Database
  ↑
Engine（内部ワーカー）
```

ポイントは3つです。

1. **DBに直接触る責任者を減らす**  
   Backend を唯一の正規入口にし、Engine は内部ワーカーとして動かします。

2. **仕様と型の正本を1つにする**  
   Frontend / Backend / Engine が別々の定義を持たず、共通のAPI仕様から揃う形にします。

3. **本番は認証必須を標準にする**  
   「公開してよいものだけ公開する」運用に変えます。

---

## 3. 何を設計から見直すべきか

### A. データの出入り口

今は Engine と Backend が同じDBを別々に更新しています。  
これを続けると、表示と内部処理で食い違いが出やすく、障害調査も難しくなります。

**今後の標準**

- DBを書き換えるのは原則 Backend
- Engine は API またはジョブキュー経由で処理する
- どこで何を更新したかを追跡しやすくする

### B. データベースのルール

今はカラム名の書き方が混在し、外部キー制約も不足しています。  
このままだと、SQLの書き間違い、結合ミス、消してはいけないデータの削除が起きやすいです。

**今後の標準**

- カラム名は `snake_case` に統一
- 外部キー、一意制約、NOT NULL を適切に設定
- 人が見る番号と、DB内部で使うIDを分ける

### C. 型と仕様

今は TypeScript 側だけに shared があり、Python Engine とはズレる可能性があります。

**今後の標準**

- API仕様を正本にする
- そこから TypeScript と Python の型を生成する
- 「画面」「API」「Engine」で同じ意味のデータを同じ形で扱う

### D. 認証と公開範囲

今はデプロイ設定に、将来的なセキュリティ事故につながる弱い箇所があります。

**今後の標準**

- 本番APIは認証必須
- 権限はロールで管理
- パスワードや秘密鍵は Secret Manager で管理
- `allow-unauthenticated` を標準にしない

---

## 4. この方針で何がよくなるか

- 機能追加しても、別の場所が壊れにくくなる
- 障害が起きたときに原因を追いやすくなる
- セキュリティ監査や将来の社外公開に耐えやすくなる
- 担当者が増えても引き継ぎしやすくなる
- 「その場しのぎの修正」を減らし、開発コストを下げられる

---

## 5. 逆に、何が重くなるか

- 最初は小さなバグ修正よりも土台整備の比重が上がる
- 一部の修正は、単独ではなくまとめてやった方が安全になる
- すぐ見える画面改善より、先に内部整理が必要な場面がある

これはデメリットというより、**短期の楽さより長期の安全性を選ぶ** という判断です。

---

## 6. 具体的な進め方

### Phase 1. 先に決める

- DB更新の正規入口は Backend に統一する
- API仕様を正本にする
- 本番は認証必須にする
- DB命名規則を `snake_case` に揃える

### Phase 2. 土台を直す

- 認証・認可
- API契約
- DB制約とマイグレーション
- Engine と Backend の接続方式

### Phase 3. 個別の不具合を直す

- `any` 型の乱用
- モックデータ残存
- UXの粗さ
- 例外処理不足
- 巨大関数や巨大ページの分割

---

## 7. この方針での判断

**答えは「全部作り直す」ではありません。**

正しくは、

**設計は見直す。実装は使えるものを残して段階的に入れ替える。**

これが、いまの judgesystem に対して最も現実的で、将来にも強い best practice です。
