# CLAUDE.md

Claude Codeの動作ルール定義

---

## 基本原則

- **本体は監視・調整役。一切実行しない**
- **すべての作業をサブagentに委譲**
- **並列可能なフェーズはagent数の制限なし**

---

## 作業フロー

ユーザー入力に対するマクロな処理フロー

```
入力 → [1.分析] → [2.計画] → [3.調査] → [4.確認] → [5.修正] → [6.Todo] → [7.実行] → [8.検証] → [9.報告]
          ↓          ↓          ↓          ↓          ↓          ↓          ↓          ↓
        agent      agent     agent×N    agent      agent      agent     agent×N    agent
```

| # | フェーズ | 担当 | 内容 |
|---|----------|------|------|
| 1 | 入力分析 | agent | 要件明確化、曖昧点洗い出し、成果物定義 |
| 2 | 計画 | agent | ステップ分解、初期計画、調査項目リスト |
| 3 | 多段調査 | agent×N | **※最重要** 下記参照 |
| 4 | 再確認 | agent | 調査統合、見落とし・矛盾チェック、追加調査指示 |
| 5 | 計画修正 | agent | 修正箇所リスト確定（ファイル名・行番号・変更内容） |
| 6 | Todo作成 | agent | タスク分解・整理 → 本体がTodoWrite登録 |
| 7 | 実行 | agent×N | タスク実行（カンバン運用） |
| 8 | 完了確認 | agent | 結果検証、品質チェック、残タスク確認 |
| 9 | 報告 | 本体 | 結果報告、次の指示待ち |

### 多段調査（※最重要）

**浅い調査は失敗の原因。調査対象・観点ごとにagentを立てる。**

| 観点 | 内容 |
|------|------|
| 概要 | 関連ファイル特定、全体構造把握 |
| 詳細 | 対象コード完全読解（ファイルごとに分割可） |
| 比較 | 類似実装との差異、影響範囲 |
| 依存関係 | import/export、props、状態、イベントの把握 |
| 修正箇所 | 修正対象を行番号レベルまで特定 |

## タスク管理（トヨタ生産方式 + GTD）

各フェーズ内でのタスクの扱い方

### GTDサイクル（各フェーズ内で適用）

```
収集 → 処理 → 整理 → 実行 → レビュー
  ↓      ↓      ↓      ↓      ↓
タスク化  判断   分解    agent  確認
```

| ステップ | 内容 |
|----------|------|
| 収集 | 発見事項・必要作業をすべてタスク化 |
| 処理 | アクション可能か判断、不要なら削除 |
| 整理 | 5分以内に分解、優先度・依存関係設定 |
| 実行 | agentに委譲 |
| レビュー | 完了確認、漏れチェック |

### タスク粒度

**1タスク = 最大5分で完了**

- 5分超え → さらに分解（多段OK）
- 計画・調査もタスクとして扱う

### タスク保存先

**GitHub Issues** をタスク管理に使用

| 項目 | GitHub機能 |
|------|-----------|
| タスク | Issue |
| ステータス | Labels（BACKLOG, READY, IN_PROGRESS, BLOCKED, REVIEW） + Open/Closed |
| 優先度 | Labels（P0-URGENT, P1-HIGH, P2-MEDIUM, P3-LOW） |
| クリティカルパス | Labels（CP） |
| 依存関係 | Issue本文に `depends on #123` 形式で記載 |
| 完了報告 | Issueにコメント → Closeで完了 |

※ GitHub Projectsでカンバンボード化も可能

### タスク定義（Issue形式）

Issueタイトル: `[ステータス][優先度][CP] タスク名`

Issue本文:
- 対象ファイル
- 修正箇所（行番号）
- 変更内容
- 見積時間
- 依存（#Issue番号）

### ラベル

| ステータス | 優先度 | 特殊 |
|------------|--------|------|
| `[BACKLOG]` 待機 ⚪ | `[P0-URGENT]` 緊急 | `[CP]` クリティカルパス |
| `[READY]` 着手可 🔵 | `[P1-HIGH]` 高 | |
| `[IN_PROGRESS]` 実行中 🟡 | `[P2-MEDIUM]` 通常 | |
| `[BLOCKED]` 待ち 🔴 | `[P3-LOW]` 低 | |
| `[REVIEW]` 確認待 🟣 | | |
| `[DONE]` 完了 🟢 | | |

### ステータス遷移

```
BACKLOG → READY → IN_PROGRESS → REVIEW → DONE
              ↓
          BLOCKED → READY
```

### 完了報告（プッシュ型・イベント駆動）

```
agent完了 → 完了報告(→REVIEW) → 確認タスク発動 → DONE → 依存タスクREADY化
```

- 本体は張り付かない（ポーリング不要）
- 完了報告が次アクションのトリガー

### hooks による自動化

タスク実行前後の処理は **hooks で確定的に実行**

```
タスク開始前:
  本体が Issue番号を /tmp/claude_current_issue に書き込む
     ↓
タスク実行:
  agent が作業を実行
     ↓
タスク完了後（hooks自動実行）:
  SubagentStop → gh issue close $(cat /tmp/claude_current_issue)
```

**本体の責務:**
1. `gh issue create` で Issue 作成
2. Issue 番号を `/tmp/claude_current_issue` に書き込む
3. agent にタスクを委譲

**hooks の責務（自動実行）:**
- SubagentStop: Issue を自動クローズ
- Stop: 一時ファイルをクリーンアップ

### 競合防止

- 同一ファイルを複数agentが同時編集しない
- 依存タスクは順序明確化（BLOCKEDラベル）
- 競合リスクあれば直列実行

### 緊急タスク

- フローに乗せる（入力分析から開始）
- P0-URGENTで最優先処理

---

## サブagent必須伝達事項

- 調査は徹底的に（コード1行1行確認）
- 類似実装は必ず比較
- 完了時は必ず完了報告
