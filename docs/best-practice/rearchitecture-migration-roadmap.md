# judgesystem 段階移行ロードマップ

**目的**: 再設計の基本方針に沿って、既存機能を止めずに段階移行する  
**前提**: 全面リライトではなく、危険な土台から順に安全化する

---

## 1. ロードマップの考え方

移行は、次の順番で進めます。

1. まず事故になりやすい箇所を止血する
2. 次にシステム境界と契約を固定する
3. その上でDBとEngineの土台を直す
4. 最後にFrontendと運用基盤を標準化する

この順番にする理由は、**土台が不安定なまま画面や細部をきれいにしても、後でやり直しになりやすい** からです。

---

## 2. 全体像

### Phase 0. 緊急是正

**目的**  
重大なセキュリティ・データ事故リスクを先に下げる

**対象Issue**

- `#103` SQLインジェクション脆弱性
- `#104` デプロイ設定の認証・認可不備
- `#106` seed.sh のカラム順序不一致

**主な成果物**

- 危険なSQL生成の封じ込め
- 本番設定の最低限の安全化
- seed とスキーマの整合確認

**完了条件**

- 明確なインジェクション経路が閉じている
- 本番公開設定に未認証の穴が残っていない
- seed で誤投入が起きない

### Phase 1. システム境界の確定

**目的**  
「どこが正規入口か」を固定し、以後の修正をぶらさない

**対象Issue**

- `#120` Engine と Backend の責任分担
- `#104` 認証ポリシーの具体化
- `#116` エラーレスポンス統一

**主な成果物**

- Backend を正規入口とする設計決定
- 本番認証必須の標準化
- API エラー形式と入力検証ルールの統一

**完了条件**

- 今後の修正で「画面から直接DB」「Engineが独自仕様で更新」が増えない
- API の入口ルールが文書化されている

### Phase 2. 契約主導への移行

**目的**  
型と仕様の正本を1つにする

**対象Issue**

- `#117` shared 型定義統一
- `#123` Python との型共有
- `#108` backend の `any` 削減
- `#111` frontend の `any` 削減
- `#116` 入力バリデーション不足

**主な成果物**

- OpenAPI または JSON Schema の定義
- Frontend / Backend / Engine の型同期ルール
- 共通のバリデーション方針

**完了条件**

- 新しいAPIは正本仕様から管理される
- 重要な入出力で手書き型の重複が減っている

### Phase 3. DB正規化と安全な移行

**目的**  
DBを将来も壊れにくい形に整える

**対象Issue**

- `#105` マイグレーションのデータ整合性
- `#106` seed 整合性
- `#121` 命名規則と外部キー制約

**主な成果物**

- 命名規則統一方針
- 内部IDと業務番号の整理
- 安全なマイグレーション手順
- 外部キー・一意制約の追加計画

**完了条件**

- 新規・更新系テーブルの標準が定義済み
- 危険な型不一致や破壊的移行が減っている

### Phase 4. Engineの分離と安全化

**目的**  
Engine を内部ワーカーとして整理し、Backend との責務を分ける

**対象Issue**

- `#120` DB 直アクセス構造
- `#112` エラーハンドリング不備
- `#113` Python アンチパターン
- `#114` 巨大関数の責務分離
- `#103` SQL安全化の仕上げ

**主な成果物**

- Engine の責務分割
- 構造化ログ
- 安全なDBアクセス方式
- Backend と連携するためのワーカー設計

**完了条件**

- Engine の主要処理が分割されている
- 危険な直接SQL・例外握り潰しが減っている
- Backend と Engine の境界が明確である

### Phase 5. Frontend標準化

**目的**  
Frontend を保守しやすい構成に寄せる

**対象Issue**

- `#109` 依存配列と競合
- `#110` モック残存
- `#111` 型安全性不足
- `#115` UX問題
- `#118` data層の責務混在
- `#122` ページ肥大化・状態管理乱立

**主な成果物**

- 共通APIクライアント
- サーバー状態管理の標準化
- モック経路の排除
- 主要ページの分割
- UXパターンの統一

**完了条件**

- 主要画面で直書き `fetch()` が消えている
- ローディング・エラー・更新の扱いが統一されている
- 本番経路のモックがなくなっている

### Phase 6. 品質ゲートと運用成熟

**目的**  
再設計後の品質を維持できる状態にする

**対象Issue**

- `#108`, `#111`, `#112`, `#113`, `#114`, `#116`, `#122` の再発防止

**主な成果物**

- テスト基盤
- CI の品質ゲート
- 監視・構造化ログ・アラート
- ドキュメント更新

**完了条件**

- 重要な変更が自動検証を通る
- 障害時に原因追跡できる
- 方針違反の変更が入りにくい

## 4. この順番を崩すと起きやすいこと

### 先にFrontendだけ整える場合

- 後からAPI仕様変更が入って作り直しになる
- 型共有がないまま画面だけきれいになる

### 先にEngineだけ分割する場合

- Backend との責任分担が曖昧なまま設計が固まる
- 将来また結合し直しになる

### 先にDB命名だけ変える場合

- API仕様、Engine、seed、マイグレーションが同時に揃わず混乱しやすい

---

## 5. 途中で確認すべき節目

### Checkpoint 1. Phase 1 完了時

- Backend を正規入口にする方針が固まっているか
- 本番認証必須の前提が合意されているか

### Checkpoint 2. Phase 2 完了時

- API仕様を正本とする運用が始まっているか
- 手書きの重複型が増え続けていないか

### Checkpoint 3. Phase 3 完了時

- DB標準が明文化されているか
- 破壊的移行の扱いが整理されているか

### Checkpoint 4. Phase 4 完了時

- Engine の責務が Backend と切り分けられているか
- 危険な直接更新が減っているか

### Checkpoint 5. Phase 5 完了時

- Frontend の主要画面が標準パターンに揃っているか
- モックや場当たりの取得方法が残っていないか

---

## 6. 最終的な到達状態

このロードマップの完了後、judgesystem は次の状態を目指します。

- Backend が正規入口として機能している
- Frontend / Backend / Engine の契約が揃っている
- DB が整合性を守る基盤として機能している
- 本番環境が認証必須・最小権限で運用されている
- Engine が分割され、保守しやすくなっている
- Frontend が標準化され、同じ不具合を繰り返しにくい

---

## 7. 実務上の進め方

このロードマップに沿う場合、実務では次の順番で管理する。

1. Phaseごとに完了条件を決める
2. そのPhaseに属する Issue をまとめて扱う
3. 方針に反する個別修正は入れない
4. 各Phaseの終了時に、次のPhaseへ進んでよいかを確認する

このやり方にすると、優先順位がぶれにくくなり、後戻りを減らせます。
