# judgesystem 全体アーキテクチャ / DB / 資源配置 承認シート

**対象**: 非エンジニア向け  
**目的**: システムの土台をどうするかを、そのままレビュー・承認できる形にする

この文書では、技術の細かい実装ではなく、**システムの基本ルール**を決めます。

- `おすすめ案で承認`
- `修正して承認`

## 項目1. システムの正規入口

**決めること**  
誰がデータを更新する正規の窓口になるか

**おすすめ案**  
Frontend も Engine も、最終的には Backend API を正規入口にする

**理由**  
データの出入り口を1つにすると、食い違いと障害調査の難しさを減らせるため

## 項目2. Engine の位置付け

**決めること**  
判定エンジンが将来もDBを直接触るかどうか

**おすすめ案**  
Engine は内部ワーカーとし、将来的にはDB直更新しない

**理由**  
裏側処理が台帳に直接書き込む構造を減らした方が、長期的に安全だから

**承認欄**

- [ ] おすすめ案で承認

- [ ] 保留

## 項目5. マスターの正式な置き場

**決めること**  
会社・拠点・発注者・工種・資格などの正式な台帳をどこに置くか

**おすすめ案**  
正式な運用マスターは PostgreSQL に置き、`data/master` や Google Sheets は入力・移行用として扱う

**理由**

**承認欄**

## 項目6. 公告文書の保存先

**決めること**  
PDF、Markdown、OCR JSON のような文書資源をどこに置くか

**おすすめ案**  
文書ファイル本体はオブジェクトストレージに置き、DBにはパスやメタ情報だけを持つ

**理由**  
大きなファイルをDB本体に抱え込みすぎない方が運用しやすいため

**非エンジニア向けの言い換え**  
「台帳の中に全部ファイルを入れる」のではなく、「台帳には置き場所だけを書き、本体は倉庫に置く」考え方です。

**承認欄**

- [ ] おすすめ案で承認
- [ ] 修正して承認
- [ ] 保留

## 項目7. 協力会社添付ファイルの保存先

**決めること**  
協力会社との送受信ファイルをどこに置くか

**おすすめ案**  
将来的には公告文書と同様に、ファイル本体はオブジェクトストレージに寄せ、DBにはメタ情報を持つ

**理由**  
ファイル保存の考え方を揃えた方が、運用と保守が簡単になるため

**補足**  
現状はDB保存ですが、best practice ではファイル倉庫を分ける方が一般的です。

**承認欄**

- [ ] おすすめ案で承認
- [ ] 修正して承認
- [ ] 保留
